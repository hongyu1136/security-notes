# Impossible

## 源码

```php
$target_file = md5(uniqid() . $uploaded_name) . '.' . $uploaded_ext;  // 随机文件名

// 先 re-encode 图片，剥离所有非图像数据
$img = imagecreatefromjpeg($uploaded_tmp);
imagejpeg($img, $temp_file, 100);
imagedestroy($img);

rename($temp_file, $target_path . $target_file);
```

| 防线 | 效果 |
|------|------|
| 后缀白名单 | `jpg/jpeg/png` |
| MIME 类型检查 | `image/jpeg` `image/png` |
| `getimagesize()` | 验证是真图片 |
| **二次编码** | `imagejpeg()` 重新生成——图片马里的 PHP 代码被剥离 |
| 随机文件名 | `md5(uniqid())`，无法预测路径 |
| CSRF token | 防跨站上传 |

## 为什么无解

**二次编码是杀手锏。** PHP 代码嵌在图片元数据里，`imagecreatefromjpeg` + `imagejpeg` 会把像素重新编码，所有非图像数据被丢弃。就算文件名可预测，上传后文件里只剩纯像素。

## 核心教训

二次编码 > 后缀校验 > MIME 校验 > 什么都没做。图片马上传后服务端重新生成图片是最稳妥的防御。
