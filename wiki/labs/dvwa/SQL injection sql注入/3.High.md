# High

## 源码 (`high.php`)

```php
$id = $_SESSION['id'];

$query  = "SELECT first_name, last_name FROM users WHERE user_id = '$id' LIMIT 1;";
$result = mysqli_query(...) or die('Something went wrong.');
//                                  ^^^^^^^^^^^^^^^^^^^^^^^^ 无声错误，不暴露数据库信息
```

## 比 Medium 多了什么

- 输入**不从 HTTP 请求直接取**，从 `$_SESSION['id']` 取值
- 查询加 `LIMIT 1` 限制输出行数
- 错误回显被屏蔽（`Something went wrong.`）

## 为什么还能注入

Session 值来自**另一个页面的用户输入**被写入 `$_SESSION['id']`。先在那个页面注入，再到这个页面触发。

## 攻击步骤

1. 在另一个输入页（`session-input.php` 或类似）输入 `1' or '1'='1`
2. 该页把 payload 写入 `$_SESSION['id']`
3. 回 SQLi 页面刷新 → 触发注入

## 受限之处

| 限制 | 导致 |
|------|------|
| 错误回显屏蔽 | 联合查询/报错注入失效 |
| `LIMIT 1` | 多行输出不可见 |

→ 只能用**布尔盲注**或**时间盲注**，逐字符猜解数据。

## 布尔盲注示例

```
1' and sleep(5)#    → 页面卡 5 秒 = 条件成立
1' and length(database())=4# → 正常返回 = 库名长度是 4
```
