# Impossible

## 源码 (`impossible.php`)

```php
checkToken($_REQUEST['user_token'], $_SESSION['session_token'], 'index.php');

$id = $_GET['id'];

// 第一道门：是否全是数字？
if (is_numeric($id)) {
    // 第二道门：强制转整数
    $id = intval($id);

    // 第三道门：PDO 预处理 + 整数类型绑定
    $data = $db->prepare('SELECT first_name, last_name FROM users WHERE user_id = (:id) LIMIT 1;');
    $data->bindParam(':id', $id, PDO::PARAM_INT);
    $data->execute();
}
generateSessionToken();
```

## 比 High 多了什么

| 防线 | 效果 |
|------|------|
| `is_numeric($id)` | 非数字输入直接拦截，页面不执行查询 |
| `intval($id)` | 浮点/十六进制/科学计数法全转成普通整数 |
| `PDO::prepare` | 参数化查询，数据和 SQL 结构彻底分离 |
| `PDO::PARAM_INT` | 参数**强制整数类型**，字符串 payload 无法注入 |
| CSRF token | 防跨站请求伪造 |

## 为什么无法绕过

```
输入: 1' or '1'='1
  ↓
is_numeric("1' or '1'='1") → false → 直接跳过查询
```

```
输入: 1 or 1=1
  ↓
is_numeric("1 or 1=1") → false → 跳过
```

```
输入: 1
  ↓
is_numeric("1") → true
intval("1") → 1
PDO bindParam → SELECT ... WHERE user_id = 1 (纯整数)
```

没有字符串拼接，就没有注入点。参数化查询 + 整数类型绑定 = **SQL 注入的终极防御**。

## 核心教训

| 只做单项 | 效果 |
|----------|------|
| 只用 `real_escape_string` | Medium 已证明——数字型注入仍生效 |
| 只用 `is_numeric` | 不够——浮点数 `1.0` 可能绕过某些场景 |
| 只用 PDO 预处理 | **已经够了**——数据和 SQL 分离是根本方案 |
| `is_numeric` + `intval` + PDO + `PARAM_INT` | ✅ 多层纵深，每一层独立有效 |

edusrc 实战中看到 PDO 不代表安全，必须确认 `bindParam` 的**参数类型**和**是否所有参数都绑定**。
