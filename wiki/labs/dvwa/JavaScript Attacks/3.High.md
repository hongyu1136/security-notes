## 源码审计

JS 高度混淆，函数名被替换成 `2y` `2B` `2J` 等。反混淆后逻辑：

```javascript
function 2y(e){...}   // 反转字符串
function 2B(e="XX"){   // 给 token 前面加 "XX"
    token.value = e + token.value;
}
function 2J(t,y="ZZ"){  // 点 Submit 时给 token 后面加 "ZZ"
    token.value = token.value + y;
}
function 2K(a,b){      // 把 phrase 反转赋给 token
    token.value = 2y(phrase.value);
}
```

token 生成时序：
```
页面加载 → 2K("XX",44) → token = reverse("") = ""
300ms后  → 2B("XX")     → token = "XX"
点Submit → 2J("ZZ")     → token = "XXZZ" ← 永远是这个值
```

但**服务器验证的公式完全不同**：

```php
$token == hash("sha256", hash("sha256", "XX" . strrev("success")) . "ZZ")
// = sha256( sha256("XXsseccus") + "ZZ" )
```

前端 JS 故意生成错误值（"XXZZ"）来混淆——你必须**绕过 JS 直接算**。

## 绕过

Console 被 `addEventListener("click")` 拦截，用 **Burp 抓包**：

1. phrase 填 `success`，开 Burp 拦截
2. 抓到 POST 请求，改 `token=ec7ef8687050b6fe803867ea696734c67b541dfafb286a0b1239f42ac5b0aa84`
3. Forward → 通过

## 关键教训

1. **JS 混淆 ≠ 安全**——混淆只增加逆向成本，不保护逻辑
2. **前端事件监听会干扰 Console 注入**——用 Burp 绕过
3. **三关总结**：客户端不可信——算法再复杂（MD5→反转→SHA256），在 F12/Burp 面前都是透明的。正确做法是服务器生成 token + 加盐。
